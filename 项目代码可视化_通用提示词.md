## 页面功能要求

### 一、树形导航

- 左侧展示与项目文件夹结构完全一致的树形导航
- 点击目录节点展开/折叠子节点
- 点击文件节点在右侧内容区展示该文件的完整分析
- **所有交互必须使用事件委托（Event Delegation），绝对禁止 inline event handler**（`onclick`、`oninput` 等），否则会出现局部变量不可访问导致点击无效的 bug

### 二、文件概述（双版本可切换）

- 每个文件展示两个可切换的概述版本，通过标签页切换，颜色区分：
  - **专有名词版**（蓝色标签）：使用原技术术语概括文件功能，包含具体技术栈、数据库表名、核心方法名
  - **新手易懂版**（绿色标签）：用通俗大白话描述，让非技术人员也能理解
- **每个文件的概述必须各自独立、基于该文件自身代码内容生成，严禁多个文件共用同一概述**

### 三、涉及知识标签

- 每个文件下方展示该文件涉及的技术知识点标签
- 按类别着色：AI/ML、数据库、搜索引擎、网络、Token、文件操作、时间、Web 等

### 四、方法卡片

- 每个方法以卡片形式展示，包含以下区域：

  **1. 方法名**

  **2. 方法功能概述**（前面加"方法功能："中文引导）
  - 一句具体的中文概述，基于方法名、参数、内部代码内容来描述该方法实际做了什么
  - **严禁使用模板化泛化描述**（如"执行模块化的业务逻辑处理""执行数据库操作"）
  - 即使两个方法功能相似，也要分别为每个方法写出针对它自己的具体描述
  - 示例：好 → "向 ftp_task_execute_log 表插入任务启动记录并返回日志ID"；坏 → "执行数据库操作"

  **3. 执行步骤列表**（前面加"执行步骤："中文引导）
  - 每步使用中文标签标注类型：`参数` / `赋值` / `执行` / `返回` / `条件` / `循环` / `异常` / `调用` 等
  - 每步用中文描述具体做了什么操作
  - **严禁任何步骤留空**——即使简单方法（如 `__init__`、`close`）也要列出具体赋值、资源释放等操作

  **4. 详细逻辑下拉区域**（前面加"详细逻辑："中文引导，可展开/收起）
  - 展示方法的完整执行描述
  - **关键操作必须贴出对应的源码原文片段**（SQL 语句、正则表达式、文件路径、API 参数等），用黄色边框等宽字体展示，不能只写抽象描述
  - 子调用标注：带 `↳` 缩进，显示被调用函数名 + 该函数概述

- 每个方法卡片内放置独立搜索框，实时过滤当前文件内的方法名，显示匹配计数

### 五、纯脚本文件处理

- 对无方法定义的纯脚本文件（只有模块级代码），展示「脚本执行步骤」卡片 + 同样享受双版本概述
- 不允许出现空白内容

### 六、Main 入口函数分析

- 对所有含 main 入口的文件（`if __name__ == '__main__'` 模块级代码块），在方法列表上方额外展示绿色 **Main Entry Flow** 卡片
- 展开后上方显示中文引导："▶ 以下是该程序的完整执行流程，展示了从启动到结束的每一步操作及函数调用关系。"
- 每步标注：
  - 函数调用跳转关系（橙色标签，如"调用 xxx() → 进入 yyy 模块处理"）
  - 条件分支 / 循环 / 异常处理等说明（紫色标签）
- **确保所有含 main 入口的文件全覆盖，一个不漏**

### 七、UI 风格

- 暗色专业主题（类 IDE 风格）
- `JetBrains Mono` 等宽字体展示代码元素
- 各功能区域明确颜色区分（概述蓝/绿、Main Flow 绿、脚本步骤橙、知识标签紫、子调用绿、源码块黄色边框等）
- 所有 UI 标签、引导说明使用中文（如"方法功能：""执行步骤：""详细逻辑：""参数""赋值""执行""返回""条件""循环""异常"等）

## 技术铁律（必须严格遵守）

1. **编码**：使用 `utf-8-sig` 读取所有文件，兼容 BOM 头文件
2. **完整性**：所有文件必须被完整解析，空文件也需在树中展示，纯脚本文件也要有内容
3. **不泛化**：每个方法概述必须基于实际代码具体描述，禁止使用模板化短语
4. **不共用**：每个文件概述必须各自独立，禁止多个文件共用同一模板
5. **不空白**：所有方法的执行步骤和详细逻辑必须有实际内容，不能留空
6. **贴源码**：SQL、正则、路径、API 参数等关键代码必须展示原文
7. **用中文**：所有操作标签、引导说明使用中文，方便英语不好的用户理解
8. **事件委托**：所有交互用事件委托，禁止 inline event handler
9. **不误删**：迭代修改时必须基于完整数据源合并重建，禁止仅用不完整的中间 JSON 文件，否则会导致已有内容（如中文概述）丢失
10. **每个方法都写**：即使两个方法功能相似，也要为每个方法写各自的具体描述，用户说过"宁愿重复也不要留空"
